# Pair Relay — 3-tier Pair Programming with Agent Handoff

通常の `pair-nav` は「人間ユーザー＝Driver / Claude＝Navigator」。
このスキルは **Driver もエージェント化** して、人間ユーザーは「PO / E2E 確認役 / 最終承認者」に回る。司令塔（このスキルを実行する親）が **Navigator サブエージェント** と **Driver サブエージェント** を交互に動かし、両者のコンテキストが満ちる前に **引継ぎ文章で別エージェントに差し替える**。

## 役割定義

| 層 | 担当 | 使うツール | 何をしないか |
|---|---|---|---|
| **User** | ゴール提示、E2E 動作確認、最終承認、方針修正 | (会話) | コードは書かない |
| **司令塔 (Orchestrator)** | Navigator/Driver の spawn、メッセージ中継、検証コマンド実行、コンテキスト監視、handoff 判定、User への報告と協力依頼 | `Agent`, `Bash`, `Read`, `Grep`, `Glob` | **自分では Edit/Write/NotebookEdit を叩かない**（Driver の仕事を奪わない） |
| **Navigator サブエージェント** | 「次の 1 件」を diff + なぜ で出す。コードのセルフレビューと、Driver 結果のレビュー | (Read 系のみ) | Edit/Write 禁止（pair-nav の原則 1） |
| **Driver サブエージェント** | Navigator の指示通りに Edit/Write を 1 件だけ実行し、結果を 1〜2 行で報告 | `Edit`, `Write`, `Read` | 自発的判断で範囲を広げない、提案しない |

司令塔自身が編集を肩代わりすると Driver の存在意義が消え、handoff の練習にもならない。**例外**: Driver が 2 回連続で同じ箇所を間違える場合のみ、司令塔が `Edit` で直接修正し「Driver スタック検出 → 介入した」と User に明示する。

## 標準ループ

```
(1) Orchestrator:  User からゴール受領
                   → 全体方針を箇条書き (3〜7 ステップ) で User に確認
                   → 承認後、Navigator を spawn

(2) Orchestrator → Navigator:
    「現在の状態 (files, last validation), 次のゴール 1 件を出して」
                   ↑                                 ↓
                   └────────  diff + なぜ  ──────────┘

(3) Orchestrator → Driver:
    Navigator が出した diff/コードをそのまま渡す
    「以下を適用してください。範囲は X のみ。完了したら触った行数だけ報告」
                   ↑                                 ↓
                   └─────────  「N 行編集完了」  ────┘

(4) Orchestrator:  検証 (cargo check / pytest / tsc 等) を自分で実行
                   失敗 → Navigator にエラー全文と該当箇所を渡してレビュー依頼 → (3) へ戻る
                   成功 → Navigator に「実コードのレビュー」依頼

(5) Orchestrator:  Navigator のレビュー結果が OK → User に 1 行報告し次ステップへ
                                       NG → 修正点を Driver に渡す → (3) へ

(6) どこかで handoff トリガーが発火 → 該当エージェントを差し替え (下記)

(7) 全ステップ完了 or User 停止指示 → 総括 2〜3 行
```

### Navigator が従うべきルール（リマインド）

司令塔は Navigator を spawn する prompt に **必ず `pair-nav/SKILL.md` を読ませる** こと。Navigator は:

- Edit/Write しない
- 1 ターン 1 作業
- 新規ファイル＝全文、既存ファイル＝diff ブロック + 各「なぜ」
- 最初から最終形を出す（後出しレビュー禁止）
- バージョン等は仮定して ship、質問ブロックしない

司令塔は Navigator の出力がこれに違反していたら **Navigator に差し戻す**（User には見せない）。

## Handoff Swap — 交代の判定と手順

### いつ swap するか（heuristics）

司令塔は各サブエージェントへの **連続ターン数** と **直近の応答長** を追跡し、以下のいずれかで swap を発火する:

| シグナル | 閾値 (目安) |
|---|---|
| 同一 Navigator への連続ターン数 | **15 ターン超** |
| 同一 Driver への連続ターン数 | **20 ターン超** |
| サブエージェントが「コンテキストが厳しい」「これまでの経緯が曖昧」を訴える | 1 回でも出たら即時 |
| 同じ提案を 2 回繰り返す / 既に決定した方針を蒸し返す | 検出時すぐ |
| `Agent` ツールが返した tool 実行が極端に遅い / トークン消費が急増 | 1 回でも出たら次の境界で |

**境界で swap する**: ステップ中盤ではなく、ある 1 件が完了して次の 1 件に移る瞬間。中途半端な状態を引き継がせない。

### handoff doc 生成

swap 直前、司令塔は退役するエージェントに **handoff doc を書かせる**（自分で書くより、当人に書かせたほうが当人視点の暗黙知が残る）:

> 「これからあなたを別の同役エージェントに交代します。後任が **これだけ読めば即座に同じ品質で続けられる** 引継ぎ文章を `references/handoff-template.md` の構造で書いてください。冗長な経緯ではなく、後任の次アクションに必要な情報だけ。」

司令塔は受け取った handoff doc を `pair-relay-workspace/handoffs/<role>-<timestamp>.md` に保存し、**新エージェントを spawn する prompt にそのまま貼る**。新エージェントは `pair-nav/SKILL.md`（Navigator なら）と handoff doc の 2 つを最初に読む。

### handoff doc の最小要素

`references/handoff-template.md` 参照。要素だけ列挙:

1. **ゴール（全体）** — 1〜2 行
2. **現フェーズ / 完了済みステップ** — 箇条書き 3〜7 項目、各 1 行
3. **次の 1 件** — 後任が真っ先にやる作業（Navigator 引継ぎなら "次に出す指示"、Driver 引継ぎなら "次に適用する diff"）
4. **触ったファイルと現状** — path + dirty/clean + 直近の検証コマンド結果（pass/fail）
5. **既知の落とし穴 / User からの制約** — 1〜3 行
6. **やらないこと（明示的に範囲外）** — 1〜2 行
7. **pair-nav 原則のうち、この場面で特に重要なもの** — 1〜2 行

「これまでの会話の議事録」は **書かない**。後任は議事録を読まずに即着手できるよう、判断結果だけを残す。

## User とのインタラクション

司令塔は User に対して以下のときだけ話しかける（チャタリングを避ける）:

1. **方針確認**: ゴール受領直後、全体ステップ案を提示し承認をもらう
2. **進捗報告**: 各ステップ完了時に 1 行 (`✅ Step 2/5: struct Foo 追加 + 検証 pass`)
3. **E2E 協力依頼**: UI / 起動が絡むテストで User の手が必要なとき。**具体的な手順** と **見るべき場所** を明示
4. **方針変更が必要なとき**: Navigator/Driver の出力から方針再考が必要だと判明したとき
5. **swap 発生報告**: 1 行で `↻ Navigator swap (15 ターン到達) — handoff: <path>`
6. **完了総括**: 2〜3 行

それ以外（サブエージェント間の往復、検証ログ、Navigator のセルフレビュー差し戻し）は **司令塔の中で閉じる**。User の画面を流さない。

### E2E 協力依頼テンプレート

UI / 起動依存テストでは:

```
🤝 E2E 確認をお願いします
- 実行: <具体コマンド or 操作手順>
- 期待: <こうなれば pass / こう見えれば OK>
- 確認すること: <チェック項目 1〜3 個、Yes/No で答えられる形>
- NG の場合: <スクショ or エラーログをそのまま貼ってください>
```

## サブエージェント spawn prompt の雛形

### Navigator を最初に spawn するとき

```
あなたは pair-relay の Navigator です。

必読:
1. .claude/skills/pair-nav/SKILL.md  ← 行動原則。Edit/Write しないこと、1 ターン 1 作業、diff + なぜ
2. (handoff doc があれば そのパス)

ゴール: <user のゴール>
完了済みステップ: <なし or 箇条書き>
触ったファイル: <list>
直近の検証状態: <pass/fail + コマンド>

最初のタスク: "次の 1 件" を出してください。
- 新規ファイルなら全文
- 既存ファイルなら diff ブロック + 各「なぜ」
- 仮定が必要なら明示して ship（質問でブロックしない）
- セルフレビュー (derive / use 位置 / マジックナンバー / 命名 / コメントの why) 済みの最終形

応答はそのまま Driver に渡せる形で。経緯の説明は不要。
```

### Driver を spawn するとき

```
あなたは pair-relay の Driver です。

役割:
- 受け取った diff/コードを Edit/Write でファイルに反映するだけ
- 範囲外の変更は禁止（提案も不要、Navigator の仕事）
- 完了したら "N 行編集完了 (path)" の 1 行で報告

タスク:
<Navigator の出力をそのまま貼る>

不明点があれば作業せず「不明: <内容>」とだけ返してください。司令塔が Navigator に差し戻します。
```

### handoff 後の後任 spawn

```
あなたは pair-relay の <Navigator|Driver> です。前任から引継ぎを受けました。

必読 (この順):
1. .claude/skills/pair-nav/SKILL.md  ← Navigator のみ
2. <handoff doc path>

最初のアクション: handoff doc の「次の 1 件」を実行してください。前任の経緯を蒸し返す必要はありません。
```

## やってはいけないこと

| アンチパターン | なぜダメか |
|---|---|
| 司令塔が痺れを切らして自分で Edit/Write | Driver の存在意義消滅、handoff 練習にならない、User への透明性が下がる |
| Navigator に「全ステップ計画して」と一気に出させる | pair-nav 原則 2 違反 (1 ターン 1 作業)、Driver も計画外行動の余地が増える |
| サブエージェント間の生のやりとりを User に垂れ流す | User がノイズに埋もれる。司令塔が要約する責務 |
| context 警告を無視して同じエージェントに 30 ターン走らせる | 終盤に精度が落ちて手戻り。早めに swap |
| handoff doc に「これまでの議論」を全部書く | 後任が読み切らず劣化引継ぎ。**次アクションに必要な情報だけ** |
| swap 後に新エージェントに `pair-nav/SKILL.md` を読ませ忘れる | 行動原則がリセットされ、勝手に Edit 始める/全文置換始める |
| Driver が「ついでにここも直しました」と範囲を広げたのを通す | サイレント編集。Navigator/User 不在の意思決定が積もる |

## 終了条件

- User が「OK、ここまでで」「自分でやる」と言った
- 全ステップ完了 + 検証 pass + Navigator レビュー OK + E2E (該当時) pass
- 方針自体の再検討が必要になり User と仕切り直し

終了時、司令塔は 2〜3 行で:
- 何を達成したか
- swap が何回発生したか（学習用）
- 残課題 / フォロー TODO（あれば）

を User に返す。
